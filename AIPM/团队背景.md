## 1. 产品愿景

**时律（Time Rhythm）** 是一款面向「时间盲视」人群（ADHD、焦虑症患者、高压职场人）的 AI 驱动视觉日程管理应用。

传统代办清单无法提供时间流逝的物理感。时律通过以下三个核心理念解决这一问题：

- **具象化**：用视觉圆环让用户「看见」时间，而非数字
- **降低门槛**：AI 自动拆解复杂任务，消除「启动困难」
- **情绪平稳**：引入能量管理，防止过度规划导致的倦怠

## 4. 核心数据模型

### Task（任务）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 主键 |
| user_id | UUID | 所属用户 |
| title | string | 任务名称 |
| start_time | datetime | 开始时间 |
| duration_minutes | int | 持续时长（分钟）|
| color | string | 圆环色块颜色（十六进制）|
| order_index | int | 排序 |
| status | enum | pending / active / done |
| energy_level | int(1-5) | 能量消耗等级（用于3.3）|
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

**重要约束：** 单日任务，不支持跨午夜（跨天任务须拆分为两条记录）。

## 6. 开发规范

### 通用
- 所有代码使用英文命名，注释可中文
- 函数不超过 50 行，文件不超过 400 行
- 禁止硬编码密钥，统一使用 `.env` 文件
- 不可变数据优先（前端 state 不直接 mutate）

### 前端
- 组件文件：PascalCase（`TimeRing.tsx`）
- Hook 文件：camelCase，`use` 前缀（`useTaskList.ts`）
- 类型定义统一放 `src/types/`
- API 调用统一通过 `src/api/` 层，不在组件内直接 axios

### 后端
- 路由层只做参数校验和调用 service，不含业务逻辑
- 业务逻辑统一在 `services/` 层
- 数据库操作统一在 `services/` 或独立 repository 层
- 所有接口须有 Pydantic schema 校验

### Git
- Commit 格式：`feat: 添加圆环拖拽功能` / `fix: 修复时间计算边界问题`
- 类型：feat / fix / refactor / docs / test / chore

## 7. 关键设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 圆环渲染技术 | SVG | 精度高、易于动画、无需 Canvas 上下文管理 |
| 实时动画 | requestAnimationFrame | 性能最优，与浏览器刷新率同步 |
| 状态管理 | Zustand | 轻量，避免 Redux 的样板代码 |
| 后端异步 | SQLAlchemy async | 高并发支持，与 FastAPI 异步模型匹配 |
| 跨天任务 | 不支持，须拆分 | 简化圆环逻辑，MVP 阶段足够 |
| MVP 同步方式 | 轮询（非 WebSocket）| 降低复杂度，WebSocket 可在后续版本引入 |

## 8. 本次开发范围（V1.0 MVP）

本次优先实现 **3.1 视觉化时间圆环**，包括：
1. 项目基础设施（认证、任务 CRUD）
2. SVG 圆环渲染（24小时映射）
3. 实时动画（当前任务色块随时间缩减）
4. 拖拽交互（调整任务时长和顺序）

**暂不实现：** 3.2 AI 任务拆解、3.3 能量监控、多设备协同、智能手表端
